#### New crate: `crates/tsk-dbg/`

Full DAP-compatible debugger connecting VS Code to TriskeleVM via TCP port 4711.

**Architecture:**
```
VS Code (DAP client)
    ↕ TCP :4711
tsk-dbg (DAP server)
    ├── dap/server.rs    — TCP framing, message dispatch
    ├── dap/types.rs     — DAP protocol structs
    ├── session.rs       — DAP handlers, spawn_blocking for VM
    ├── vm_runner.rs     — Self-contained VM interpreter with debug hooks
    ├── disasm.rs        — Inline disassembler (PC ± N instructions)
    └── symbols.rs       — .sym file loader (label → VM address)
```

**CLI:**
```cmd
cargo run -p tsk-dbg -- src\wolf3d_v2_clean.tvmx \
    --symbols src\wolf3d_v2_clean.sym \
    --max-instructions 10000 \
    --verbose
```

**Options:**
- `--symbols <file>` — .sym symbol table (generated by tsk-asm --emit-symbols)
- `--max-instructions N` — pause after N instructions (0 = unlimited, default)
- `--port N` — TCP port (default: 4711)
- `--mem-size N` — VM memory in bytes (default: 64MB)
- `--verbose` — enable DEBUG logs

## 3. Bug Fixes Discovered via tsk-dbg (v0.5.0)

These bugs were found by debugging `wolf3d_v2_clean.tasm` with tsk-dbg.

### 3.1 tsk-asm: Jump offset off by 4

**File:** `crates/tsk-asm/src/assembler.rs`, `fixup_labels()`

**Bug:** `delta = target_vm - instr_vm` — missing the +4 for PC-after-fetch.

**Fix:**
```rust
// Before (wrong):
let delta = target_vm as i64 - instr_vm as i64;
// After (correct):
let delta = target_vm as i64 - (instr_vm as i64 + 4);
```

**Impact:** ALL conditional and unconditional jumps landed 4 bytes too far.
This caused `norm_loop` to skip `V_CmpI` entirely, looping forever.

### 3.2 tsk-dbg vm_runner: D_MovI used imm14 instead of imm19

**File:** `crates/tsk-dbg/src/vm_runner.rs`

`D_MovI` encodes its immediate on 19 bits (bits [18:0]) in `tsk-asm`.
`tsk-dbg` was reading only 14 bits → `229376 (0x38000)` decoded as `0`.

**Fix:** Use `imm19 = sign_extend19(word & 0x0007_FFFF)` for `D_MovI` and `A_PushI`.

### 3.3 tsk-dbg vm_runner: V_CmpI read wrong register field

**File:** `crates/tsk-dbg/src/vm_runner.rs`

`V_CmpI` is Type I — `encode_i` places the register in `dst` field [23:19].
`tsk-dbg` was reading `src1` field [18:14] → always compared R0 (=0).

**Fix:** `let a = rd!(dst) as i64;` instead of `rd!(src1)`.

### 3.4 tsk-dbg vm_runner: Conditional jumps used imm14 instead of off24

**File:** `crates/tsk-dbg/src/vm_runner.rs`

All conditional jumps (`F_Jz`, `F_Jnz`, `F_Jl`, `F_Jle`, `F_Jg`, `F_Jge`, `F_Loop`)
used `imm14` (14-bit) instead of `off24` (24-bit) for the branch offset.

**Fix:** Replace `imm14` with `off24` in all 7 conditional branch opcodes.

### 3.5 tsk-dbg vm_runner: V_CmpI used imm9 instead of imm14

`V_CmpI` immediate is 14-bit signed. Was using 9-bit → `360` decoded as `-152`.

**Fix:** `let b = imm14;` instead of `imm9`.

### 3.6 tsk-dbg: Bytecode loaded at address 0 instead of LOAD_BASE

`triskele-vm` loads code at `NULL_PAGE_END = 0x1000`. `tsk-dbg` was loading at 0.
The `.tvmx` format has a header — must parse via `TvmFile::load_from_reader`.

**Fix:** Parse `.tvmx` via `TvmFile`, load rodata at `0x1000`, code immediately after
(4-byte aligned). All symbol addresses offset by `LOAD_BASE = 0x1000`.

## 9. Debugging Lessons (v0.5.0)

The debugging session uncovered 6 bugs across tsk-asm and tsk-dbg.
Key insight: **a debugger debugs its own interpreter** — the Ouroboros paradox
that the TSCG Ontology Diagnosis Pipeline was designed to address.

The most impactful bug was the **jump offset (-4)** in tsk-asm: all jumps
landed 4 bytes too far, causing infinite loops that looked like comparison
failures. This was invisible without the debugger's step-by-step trace.

The debugger also proved the **ISA encoding discipline**: D_MovI uses 19-bit
immediates (not 14-bit), conditional jumps use 24-bit offsets (not 14-bit),
V_CmpI reads from the `dst` field (not `src1`). These are architectural
decisions that must be documented and enforced by SHACL at M0 level.

### 10.2 JIT Compilation with Cranelift

**Motivation:** Wolf3D DDA loop is the obvious first hot path (~50% CPU).
DOOM will require more throughput. Long-term: C compiler requires near-native speed.

**Approach:**
- `cranelift` (Wasmtime's backend) — pure Rust, no LLVM dependency, Apache license
- Detect hot loops (execution counter per basic block)
- Compile hot paths to native x86/ARM at runtime
- Fall back to interpreter for cold code

**Advantage over LLVM:** lighter dependency, simpler integration,
already used in production (Wasmtime, Cranelift-JIT).

- The 16 M3 primitives (ASFID + REVOI + TKSL+poles) are the **generators**
